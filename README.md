## How AI PM interviews differ from classic PM

Classic PM rounds still apply (product sense, execution, analytics), but AI PM/Product Engineer loops add:

- **AI product judgment** — when AI helps vs hurts UX; where to use LLMs.
- **Technical fluency** — prompts, RAG, evals, latency/cost; you don't code the model but you reason about it.
- **Evals & quality** — how you'd measure an AI feature's success.
- **Data & feedback loops** — using outputs to improve the product.

---

## Questions by category

### AI product sense
- Design an AI feature for [product]. Why AI, and where would it fail?
- When should a product *not* use an LLM?
- How do you handle non-deterministic outputs in a user-facing feature?
- How do you set user expectations when the AI is sometimes wrong?

### Evals & metrics
- How would you measure success for an AI feature beyond "accuracy"?
- A new model version changes outputs — how do you decide to ship?
- Define guardrails for a customer-facing AI assistant.

### Execution
- Scope an MVP of an AI feature in 2 weeks — what's in/out?
- How do you prioritize fixing hallucinations vs adding capabilities?

### Technical fluency
- Explain RAG to a non-technical stakeholder.
- Prompt vs RAG vs fine-tune — when each, in product terms.
- What drives LLM cost and latency, and how do you manage them?

### Classic PM (still tested)
- Product sense, prioritization, metrics, stakeholder conflict, a product you love and would improve.

## Prep plan

1. Ship one small AI feature (even a prototype) so you can speak from experience.
2. Practice framing **AI trade-offs** for both technical and non-technical audiences.
3. Prepare an **evals story** — how you'd know the feature works.
4. Run an **AI PM mock interview** on [Landed](https://landed.b100x.ai).
